## جدول 2: بررسی اصول SOLID

| اصل | | کلاس | علت برقراری/نقض |
|-----|---------|------|-----------------|
| SRP | مورد برقراری | همه کلاس‌ها | هر کلاس فقط یک مسئولیت را بر عهده دارد |
| | مورد نقض | - | |
| OCP | مورد برقراری | در بقیه کلاس‌ها غیر دو کلاس گفته شده در نقض | وابستگی یا شرطی وجود ندارد پس رعایت شده |
| | مورد نقض | ReservationService, Reservation | زیرا در آن باید روش پرداخت و ارسال مسیج بررسی شود و اگر مورد جدید اضافه شود نیازمند تغییرات است و در این دو کلاس به کلاس‌های سطح پایین‌تر وابستگی وجود دارد |
| LSP | مورد برقراری | همه کلاس‌ها | همه فرزندان می‌توانند به جای کلاسی که از آن ارث بری کرده‌اند استفاده شوند و تمام رفتار پدر خود را به درستی و مشابه پیاده‌سازی کردند |
| | مورد نقض | - | |
| ISP | مورد برقراری | همه کلاس‌هایی که از اینترفیس ارث بری کردند | همه این کلاس‌ها از اینترفیس‌های کوچک و مختص به کارهای مشخص و کوچک ارث بری کردند |
| | مورد نقض | - | |
| DIP | مورد برقراری | بقیه کلاس‌ها | به کلاس‌های سطح پایین وابستگی‌ای ندارند |
| | مورد نقض | Reservation, ReservationService | در کلاس اول از کلاس‌های دیگر مثل customer و room استفاده شده و وابستگی وجود دارد، در کلاس دوم با وجود اینکه از اینترفیس برای استفاده از payment یا message sender استفاده شده نه اینکه مستقیماً آبجکت ساخته بشه از فرزندانش ولی باز از کلاس‌های دیگه استفاده کرده و به کلاس‌های سطح پایین وابستگی دارد |
| PLK | مورد برقراری | بقیه کلاس‌ها | چون نهایتاً از اطلاعات و ویژگی‌های موجود در یک کلاس دیگه استفاده می‌کنند نه زنجیره‌ای از آنها |
| | مورد نقض | ReservationService | در این کلاس برای دسترسی داشتن به شماره موبایل و ایمیل باید به مشتری و reservation دسترسی داشته باشیم و این یک زنجیره اطلاعات تشکیل می‌دهد که این مورد را نقض می‌کند |
| CRP | مورد برقراری | Reservation, ReservationService | در این دو کلاس به جای ارث بری از همدیگر از آبجکتی از کلاس اول در کلاس دوم استفاده شده و این اصل برقرار شده و باعث انعطاف بیشتر شده |
| | مورد نقض | - | |
